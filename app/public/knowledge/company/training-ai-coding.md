# AI 超级进化：从 AI 编程入门到构建个人 AI-SOP

【研发必学】AI编程能力跃迁实战课课程回顾材料，附件引用自《技术干货》Wiki 页面「AI研发实战课」板块。全篇十三部分，讲清楚从零散提问到构建个人 AI-SOP（标准作业流程）的完整能力跃迁路径。

## 0. 这门课要解决什么问题

很多人已经开始用 AI 写代码、查资料、做总结，但真正遇到的问题很集中：会零散提问但不会稳定协作；会拿到答案但拿不到可复用的方法；会用单点工具但不会把工具接成工作流；会追求一时效率却没有把经验沉淀成长期资产。

这门课要完成三步升级：从"会用 AI"升级到"会和 AI 协作"；从"单次提问"升级到"可复用方法"；从"工具使用者"升级到"个人 AI 工作方式设计者"。

**AI 改变研发角色的四层跃迁**（从工具到操作系统，变化的不只是速度，更是协作方式与责任分工）：

1. **工具层**：代码补全、语法解释、片段生成。重点：快，但只解决局部问题。
2. **助手层**：读代码、拆方案、写测试/写文档。重点：开始接管信息处理与初步判断。
3. **协作者层**：执行任务、调工具、留证据。重点：从聊天进入真实工作流。
4. **操作系统层**：Prompt / Pattern / Skill / MCP / AI-SOP。重点：把能力资产化，形成长期复利。

研发角色的变化方向，是从手工执行转向协作系统设计与流程控制。

## 一、AI 认知升级与研发角色重构

### AI 为什么正在改变研发工作方式

AI 对研发的冲击不只来自代码生成速度，而是来自"信息处理方式"的变化。过去研发主工作流是：自己读文档、自己查代码、自己拆需求、自己写方案、自己执行验证。AI 进入后，链路被拆成两层：人负责定义方向、边界、验收和取舍；AI 负责加速理解、生成、检查、汇总和回流。

变化真正发生在三个地方：信息读取速度被放大、方案生成速度被放大、试错与反馈迭代速度被放大。

以修线上 Bug 为例：
- **过去的做法**：工程师先自己翻日志、找调用链、猜问题点、写修复，再补文档。
- **现在更高效的做法**：先让 AI 归纳日志异常模式和调用链，再让 AI 给出 2-3 个故障假设，再让 AI 按"修复方案/风险/验证/回滚"结构整理候选路径。

真正被改变的，不是某一步是否自动化，而是整条链从"纯手工串行"变成"人做判断，AI 做加速"。

### AI 在研发中的角色：工具、助手、协作者

| 角色 | 典型表现 | 人的控制要求 |
|---|---|---|
| 工具 | 代码补全、查语法、生成片段 | 低 |
| 助手 | 帮你读代码、拆方案、写测试、写文档 | 中 |
| 协作者 | 能在上下文里执行任务、调工具、跑闭环 | 高 |

真正需要培养的不是"会不会用工具"，而是"知道什么时候把 AI 提升为助手，什么时候允许它成为协作者"。

### AI 能替代什么、增强什么、不能替代什么

- **更容易被替代**：机械改写、重复总结、模板化代码和文档产出、第一轮信息归纳。
- **更适合被增强**：需求拆解、方案比较、Bug 排查、测试覆盖和边界补全、文档与复盘沉淀。
- **短期内很难替代**：跨团队协调、业务优先级判断、架构取舍中的责任决策、风险承担与最终验收。

例如需求是"给订单系统加一层幂等保护"：AI 可以先列常见幂等实现方式、接口改动点、风险点和测试点，也可以生成控制器、服务层和测试样例的初稿；但是否采用数据库唯一索引、幂等表还是缓存令牌，仍然取决于业务吞吐、失败成本和团队维护习惯。

### 常见协作误区与高频失败原因

高频失败通常不是因为模型不够强，而是协作方式错误：把模糊任务直接丢给 AI；没有给边界和验收标准；只要答案不要证据；把一次成功误当成长期方法；把工具体验误当成工程体系。

### 从"偶尔使用"到"主动设计协作"

当你开始主动设计协作而不是零散提问，研发工作方式就会变：从问问题升级为设计任务；从拿答案升级为拿结果；从一次性 Prompt 升级为长期复用方法。

**从单次提问到长期复用的能力阶梯**：
- **Prompt**：解决一次问题，强调背景、目标、约束和输出。
- **Pattern**：解决一类问题，把输入输出结构与控制方法固定下来。
- **Skill**：解决一段稳定流程，绑定规则、脚本、门禁和参考材料。
- **AI-SOP**：解决一条长期工作链，让协作、验证、回流都变成可复用资产。

本章核心不是"AI 很强"这种泛泛认知，而是一个更准确的判断：AI 正在重写研发协作链路。判断 AI 用得好不好，不要只看它能不能写出一段代码，而要看是否已经开始主动设计任务、控制过程、验收结果、沉淀经验。

## 二、AI 编程基础入门

### 什么是 AI 编程

AI 编程不是"用 AI 写代码"这么简单，更准确的定义是：把模型能力嵌入研发交付链路，让 AI 参与任务理解、方案生成、执行落地、质量验证和结果沉淀。

**常见使用场景**：阅读陌生代码库、需求分析与技术方案拆解、代码生成/重构和补全、Bug 排查与异常解释、测试用例/边界条件与回归检查、文档/周报/复盘和知识沉淀。不要把 AI 只当成"写代码工具"，它覆盖的是整条研发任务链。

### 如何向 AI 描述任务

对研发最稳的任务描述，至少要包含五项：背景（当前项目、模块、上下文）、目标（想完成什么结果）、约束（不能做什么、必须满足什么）、输出（希望按什么格式交付）、验收（什么结果算完成）。最常见的问题不是"不会描述"，而是描述里缺了关键项。

**Prompt 的基础结构（五段式模板）**：
```
背景：当前项目、代码位置、已有事实
目标：希望这轮输出完成什么
约束：禁止项、边界项、环境项
输出：Markdown / JSON / 表格 / 步骤
验收：完成标准、必须覆盖项、失败定义
```

示例："帮我优化这个接口" → 更稳的写法：
```
背景：这是 Go 服务里的订单查询接口，当前 p95 延迟约 900ms。
目标：给出 2 个优化方向，并优先选择不改数据库 schema 的方案。
约束：不能改变接口返回字段；不能引入新中间件；必须考虑回滚。
输出：请按"问题判断/优化方案/风险/验证/回滚"五段输出。
验收：必须指出最可能的瓶颈位置，并给出可执行的验证步骤。
```

### 为什么 AI 输出不稳定

核心原因有三类：输入描述不稳定、上下文边界不稳定、输出验收不稳定。很多人把这归结为"模型随机"，但工程里更大的波动来自输入控制不足。

**面向研发的基础提问范式**：
- 代码阅读型："请先说明模块职责，再解释关键调用链，最后指出不确定点。"
- 方案拆解型："请给出 2-3 种实现路径，并比较复杂度、风险与回滚方式。"
- 执行型："请先列出改动计划，再按步骤实现，并提供验证结果。"
- 复盘型："请把这次问题整理为原因、影响、修复、预防四段输出。"

### 提示词为什么有效

如果把模型看成"在给定条件下继续生成"的系统，提示词真正改变的是条件项，同时影响四件事：
1. **改变条件分布**：背景、目标、约束和格式会改变后续输出的概率分布。
2. **改变注意力焦点**：标签化结构会让模型更稳定地关注关键字段。
3. **命中对齐后的任务模板**：角色、边界、输出格式和验收项足够清晰时，模型更容易进入训练中高先验的"可执行任务模式"。
4. **激活上下文学习**：少样本示例不是在"灌输知识"，而是在给模型提供可模仿的输入输出轨道。

所以提示词不是咒语，而是条件控制。

### 从提示词工程到模式工程

当任务开始跨模型、跨场景、跨时间复用时，只靠"写好一句 Prompt"很快不够，需要继续向上抽象：Prompt（解决一次任务）→ Pattern（解决一类任务，可复用模式）→ Schema（固定输入输出结构，减少格式漂移）→ Eval（用评测而不是感觉判断是否变好）→ Function/Workflow（把模式和结构打包成可调用、可编排、可版本化的单元）。

以"生成接口变更评审结论"为例逐步升级：
- Prompt：让模型先输出一次评审意见。
- Pattern：固定成"改动摘要/风险/测试/回滚"四段。
- Schema：固定成 JSON 字段：`summary`、`risks`、`tests`、`rollback`。
- Eval：检查是否漏风险、是否格式可解析、是否出现无证据结论。
- Function/Workflow：把输入参数收敛成模块名、改动文件、变更摘要、测试结果。

本章最重要的收获是建立一个习惯：以后每次和 AI 协作，都先把任务说明书写完整。会不会问，本质上不是语言技巧，而是任务定义能力。

## 三、当前主流 AI 编程工具与选择建议

### 工具谱系三分法

截至 2026 年 3 月，AI 编程工具不再只有"编辑器里补全代码"一种形态，更常见的是三类：

1. **IDE 内嵌型**：重点是补全、对话、局部改写、代码库问答。代表：Cursor（AI 原生编辑器体验）、Windsurf（强调连续工作流）、GitHub Copilot（覆盖最广的 IDE 侧产品）。适合日常补全、局部函数改写、小步快跑式迭代。优点是低门槛，上限取决于能否稳定理解整个代码库、能否真正接入执行验证和收口闭环。

2. **仓库/平台型**：重点是"在代码平台里帮你推进任务"而非"陪你写代码"。代表：GitHub Copilot coding agent，可以围绕 Issue、PR 和 GitHub 工作流在后台工作。适合从 Issue 直接起任务、代码评审、PR 草稿、后台任务推进，但更偏平台链路，不一定适合所有本地细碎研发动作。

3. **终端 Agent 型**：更接近真实工程闭环。代表：
   - **Claude Code**：强项是终端内协作、深度代码库理解、直接编辑文件、运行命令、接 MCP，适合复杂多步任务。
   - **Codex**：强项是多工作面协作，既有终端能力，也有更强的 agent 化工作流、Skills、Automations 和多代理并行能力。官方场景：重构老代码、编写和修复测试、处理 on-call 与故障检查、编写文档并保持开发者专注。

它们和 IDE 内嵌工具最大的差异，不在"谁会补全代码"，而在：能不能直接进工程环境、能不能跑命令和验证、能不能处理多步骤任务、能不能沉淀成长期协作资产。

### 常见工具对比

| 工具 | 主要形态 | 更适合 | 典型强项 | 相对短板 | 长期 AI-SOP 主力适配度 |
|---|---|---|---|---|---|
| Cursor | IDE 内嵌型 | 日常编码、局部改写、快速试方案 | 编辑器体验流畅，局部改写和规则配置方便 | 更偏编辑器主场，深度闭环能力取决于外围搭配 | 中 |
| Windsurf | IDE 内嵌型/agentic IDE | 希望在 IDE 内保持连续协作的人 | 上下文感知、Chat/Command/Autocomplete 组合完整 | 仍偏 IDE 主场，复杂工程闭环通常还要补外部流程 | 中 |
| GitHub Copilot | IDE + GitHub 平台型 | 团队已深度使用 GitHub 的场景 | 覆盖广，补全成熟，能进入 PR/Issue 流程 | 本地工程深度协作通常不如终端 Agent 自由 | 中 |
| GitHub Copilot coding agent | 平台型后台 Agent | 围绕 Issue、PR、评审做后台推进 | 能在 GitHub 流程中后台完成任务并发起 PR | 更依赖 GitHub 平台链路，不是所有本地动作都适合 | 中 |
| Claude Code | 终端 Agent 型 | 读码、改文件、跑命令、复杂多步任务 | 终端原生、代码库理解强、可接工具和多表面协作 | 需要团队本身有较好的工程规则和约束，效果才更稳 | 高 |
| Codex | 终端 Agent 型/command center | 任务闭环、多任务并行、Skills、Automations | 多代理并行、Skills、Automations、工作区和系统装配能力强 | 更适合把协作体系做深，学习投入也会更高 | 高 |

一句话总结：**IDE 型工具更适合"快"，终端 Agent 型工具更适合"深"**。

### 怎么选主力工具

按工作方式而非流行度选择：主要想更快补全/更快局部改写 → 先看 IDE 内嵌型；主要想把 AI 接到 GitHub 流程里 → 先看平台型；主要想让 AI 真正参与研发闭环 → 优先看终端 Agent 型。一个更实用的组合方式：编辑器里保留一个 IDE 型工具负责高频、小步、局部动作；主力协作放在终端 Agent，负责读码、实现、验证、审查和收口。

**现阶段更推荐重点掌握：Codex 和 Claude Code**——因为课程后半段方法论（Prompt 工程、Skill 固化、MCP 接入、AI-SOP 设计）都更接近终端与真实工程环境的能力形态：

- **Claude Code 更适合什么人**：已经很习惯终端和本地工程环境；最常做读码、排障、实现、验证这种连续多步任务；希望 AI 先成为一个很强的工程协作者。
- **Codex 更适合什么人**：不只想让 AI 做一次任务，还想形成长期工作系统；想把 Skills、Automations、多任务并行和 agent 化编排一起用起来；希望 AI 最终进入更完整的个人或团队 AI-SOP 体系。

最稳的学习路径：先用一个 IDE 工具建立日常习惯 → 再用 Claude Code 或 Codex 建立工程闭环能力 → 最后把主力沉淀到自己的 Skill 和 AI-SOP 体系里。

## 四、研发场景中的核心 AI 使用方法

### 代码阅读与项目理解

在陌生项目里最值钱的不是"AI 能不能解释一行代码"，而是"AI 能不能快速建立全局理解"。最有效方式：先识别入口 → 再识别目录结构 → 再识别关键数据流 → 最后才钻复杂细节。

可直接使用的读码 Prompt：
```
请帮助我快速建立这个项目的全局理解，按下面顺序输出：
1. 入口文件和启动链路
2. 主要目录职责
3. 请求从入口到核心业务再到持久层的关键路径
4. 你目前不确定、需要我补充的信息
要求：先讲全局，再讲局部；不要直接陷入单个函数细节；如果信息不足，请明确标记"不确定"
```

### 需求分析与技术方案拆解

AI 很适合做第一轮拆解，但不要让它直接决定方案。更合理的方式：先让 AI 列方案 → 再让 AI 对比方案 → 最后由人做取舍。

```
需求：给后台增加"导出近 30 天异常订单"的能力。
请输出：1. 需求理解 2. 2-3 种实现方案 3. 每种方案的改动点 4. 风险与回滚方式 5. 推荐方案
约束：不能引入新基础设施；优先复用现有查询接口和权限体系
```

### 代码生成、补全与重构

代码生成适合解决"怎么实现"；重构更适合解决"如何在不改业务目标的前提下整理结构"。真正要避免的，是把"临时能跑"误当成"已经可交付"。重构任务里要主动限制范围：

```
请对这个 Service 做重构，但必须遵守：
1. 不改变对外接口签名 2. 不修改业务逻辑结果 3. 只处理重复逻辑抽取和命名清晰化
4. 输出时先给重构计划，再给代码改动点，再给验证方法
```

### Bug 排查与问题定位

AI 在 Bug 场景中的价值：快速梳理调用链、快速生成假设列表、快速补边界测试、快速整理问题报告。

```
现象：支付回调接口偶发 500，日志里出现 timeout 和 nil pointer。
请按下面结构排查：1. 现象归纳 2. 最可能的 3 个故障假设 3. 每个假设对应的验证方式
4. 如果成立，修复思路是什么 5. 哪些点仍然缺证据
要求：先列假设，再列修复；不要直接把推测写成根因
```

### 测试用例、边界条件与代码校验

当团队开始把 AI 用到测试与校验，价值才真正开始稳定化，因为验证环节天然更适合标准化与复用：

```
请为这个"优惠券核销"函数补测试，至少覆盖：1. 正常核销 2. 重复核销 3. 已过期优惠券
4. 并发下重复请求 5. 空参数和非法参数
输出要求：先列测试点，再给测试代码，最后说明哪些边界仍然没法仅靠单测覆盖
```

### 文档、注释、周报与复盘输出

文档不是 AI 的"附属产物"，而是经验沉淀的主要入口。很多组织用 AI 只提升了写代码速度，却没有提升知识回流速度，结果很快遇到复用瓶颈。周报和复盘不建议直接说"帮我写一份总结"：

```
请把本周支付模块工作整理成周报，分 4 部分：1. 完成事项 2. 风险与阻塞 3. 下周计划 4. 需要协同的事项
要求：只基于我提供的事实整理；不要扩写未发生的成果；每项尽量控制在两句内
```

**项目级 AI-SOP 执行闭环**（从需求澄清到规则沉淀，执行结果必须回流到下一轮协作）：
1. 需求澄清与边界确认 → 2. 结构化拆解任务 → 3. 实施改动并验证 → 4. 风险审查关口（发现阻塞项时必须回到实现阶段修正）→ 5. 文档沉淀与交接 → 6. 提交与发布 → 7. 规则/Skill 反馈更新（阻塞→修复→复检，反馈结果进入下一轮任务）

闭环说明：审查关口失败时，流程必须回到实现环节，而不是在发布后被动修补；每一轮完成后都要留下文档、日志、规则或 Skill 更新，形成可复用资产；只有把执行结果回流到规则层，AI 协作才会越来越稳，而不是停留在单次成功。

**典型案例**：GitHub Copilot 官方研究显示 55% 更快完成任务，说明 AI 能显著提升局部开发效率与主观专注体验（课堂用法：先讲"更快"，再引出"更快不等于更稳"，自然过渡到 Skill 与 AI-SOP）；OpenAI Codex 官方场景说明 AI 如何进入任务执行闭环。

本章核心：每一类研发任务都可以被拆成更清晰的输入、过程和输出。只要把场景说清楚、把结构立住、把证据和验收补齐，AI 就不再只是回答问题，而会开始参与交付。

## 五、让 AI 输出更稳定、更可控

### Prompt 进阶：任务分层与上下文管理

进阶重点不是"写得更长"，而是"把复杂任务拆成更容易控制的层"。先分三层：问题层（这次到底要解决什么问题）、方法层（应该按什么路径推进）、结果层（最后要交付成什么格式）。复杂任务再拆成五段：任务目标、上下文信息、数据与约束、输出要求、反馈迭代。

复杂研发任务示例（五层写法）：
```
【任务目标】请判断这个缓存穿透问题最可能的成因，并给出修复方案。
【上下文信息】服务：用户画像服务；现象：某接口 QPS 突增，Redis 命中率明显下降；已知事实：最近发布过参数校验调整
【数据与约束】不能直接关闭缓存；不能修改接口协议；需要考虑回滚
【输出要求】按"判断/假设/修复/验证/回滚"输出
【反馈迭代】如果证据不足，请先列需要补的日志和指标
```

### 结构化输出：Markdown、JSON、表格、步骤化结果

结构化输出的意义不只是"看起来整齐"，而是让结果能继续被工具、脚本、工作流消费。没有结构化输出，就很难稳定进入 Agent 或 MCP 链路。

四类常见结构化输出：Markdown（讲解、方案比较、复盘和评审记录）、JSON（脚本消费、字段校验、下游自动化处理）、表格（比较多个方案/风险/任务状态）、步骤化结果（执行计划、排障路径、验证清单）。

如果希望输出稳定，最好在 Prompt 中直接把字段写死：
```
请输出 JSON，字段固定为：
{"problem_summary": "", "possible_causes": [], "recommended_action": [], "verification_steps": [], "rollback_plan": []}
要求：不要新增字段；没有信息时返回空数组，不要省略字段
```

### 如何给 AI 设置角色、边界与验收标准

角色用于收敛视角，边界用于限制发散，验收用于控制结果质量。缺任何一个，都会让输出回到"像是有道理，但不能直接用"的状态。一个稳定写法包含四项：角色（例如"以后端架构评审者的视角分析"）、边界（例如"不修改数据库 schema，不假设未提供的外部系统能力"）、输出格式、验收标准。高风险任务还建议显式加入负向约束：不要编造不存在的依赖、不要省略失败条件、不要把未经验证的假设写成事实、不要输出与约束冲突的建议。

### 如何让 AI 输出更贴近工程要求

多给真实上下文，少给抽象空话；多给失败定义，少给空泛目标；多要证据和检查项，少要漂亮结论。工程上还要注意：使用标签和分隔符把背景、约束、任务、输出格式分区；控制指令噪声（指令不是越多越好，越精确越有效）；让上下文增量输入（多轮协作时告诉模型"上轮哪里失败了，这轮只改哪里"）。

随着基座模型能力增强，纯提示词微调的收益会逐步收敛，主要瓶颈会从文案本身转向：上下文是不是真的、任务定义是不是完整、评测和门禁是不是严谨。更合理的资源分配：20% 放在提示词局部优化，30% 放在真实上下文接入，30% 放在评测体系建设，20% 放在工作流与函数化封装。

### 从"能回答"到"可复用"的控制方法

真正决定长期产出的，不是某一次 Prompt 写得多漂亮，而是这次有效控制能不能被沉淀下来。演化路径：Prompt（解决单次任务）→ Pattern（把单次成功抽象成可复用模式）→ Schema（把输入输出固定成稳定结构）→ Eval（用评测验证这个模式是否真的更好）→ 模板资产（把验证通过的模式沉淀成带元数据的模板资产）→ Function/Workflow（把模板资产进一步函数化、工作流化）→ Skill（把高频稳定能力固化成可调用单元）。

**让 AI 输出更稳定的控制回路**：背景（项目、事实、上下文）→ 目标（要完成什么结果）→ 约束（边界、禁区、环境）→ 输出（Markdown/JSON/表格）→ 验收（覆盖项、检查项、失败定义）→ 评测（结果回流与单变量迭代）。稳定输出 = 明确输入 + 可验证结果 + 失败回流。

### 先定义优化目标，再改 Prompt

很多 Prompt 优化做不出结果，是因为优化目标一开始就没定义清楚。真正有效的优化应先回答四个问题：这次要提升什么（质量/稳定性/速度/成本）、这次不能牺牲什么（如字段完整率、风险召回率、格式可解析性）、失败长什么样（如漏字段、编造事实、格式漂移、结论无证据）、成功怎么衡量（如一次通过率提升、返工轮次下降、解析失败率降低）。

**案例：Structured Outputs**——OpenAI 官方强调，结构化输出让模型结果可靠匹配预定义 schema，是工具调用、自动化流程和 Agent 链路的关键前提。

### 高质量提示词的八条工程原则

1. **明确任务和目标**：写成四元组——任务目标、约束边界、失败定义、成功定义。
2. **结构化设计与标签化**：最稳骨架之一是 `Role -> Task -> Constraints -> Output Format -> Acceptance Criteria`。
3. **使用分隔符隔离上下文**：常见写法用 `###`、`---`、XML 标签。
4. **上下文增量与反向反馈**：每轮都带上上一轮失败点和本轮增量目标，失败样本回收入评测集。
5. **少量但精确的指令**：用数量、长度、格式、禁区做硬限制，高风险任务里明确加负向约束。
6. **多维度嵌套设计**：复杂任务采用"概述 -> 维度 -> 输出格式"结构。
7. **优化提示词与动态调整**：每轮只改一个核心变量，避免失去归因；旧版和新版必须在固定环境里做 A/B 对比。
8. **系统化与参数化设计**：高频 Prompt 最终都要沉淀成模板资产，维护适用任务/失效场景/模型版本/最近评测日期等元数据。

高频骨架句式：
```
Role: 你现在扮演……
Task: 本轮要完成……
Constraints: - 不要…… - 必须……
Output Format: - 1. - 2. - 3.
Acceptance Criteria: - 必须覆盖…… - 不能遗漏……
```

### 传统技巧仍然有效，但要用工程方式使用

分步提示（让模型按固定步骤输出）、结构化提示（用 XML 或标签分区）、分隔符与上下文分区、Few-shot（给 1-3 个高质量样例，优先覆盖最容易犯错的边界情况）、负向约束（显式写"不允许做什么"）。这些不是过时方法，而是现代评测体系里的基础层。

**旧时代技巧增强包**：Few-shot 强化（样例要满足语义贴近、边界覆盖、输出锁定三个原则，2-4 个样例通常是性价比区间）；对比式提示（同时给正确样例、错误样例和错误原因，比只给正例更能压低越界率）；骨架先行（先要求输出固定骨架，再填充内容，降低长输出后半段漂移）；检查点提示（在同一任务中插入阶段自检点）。结论：真实上下文接入解决"输入是不是真的"，提示词优化解决"推理和输出稳不稳"，两者是互补关系，不是替代关系。

### 怎么判断提示词真的变好了

不能靠感觉判断，要靠评测。评测集至少覆盖四类：需求澄清集（检查边界定义与验收项完整性）、实现任务集（检查一次通过率与回滚率）、审查任务集（检查高风险问题召回率与误报率）、文档任务集（检查结构合规率与信息缺失率）。执行方式必须一致：同一模型版本、同一工具权限、同一温度参数、同一数据样本，旧版和新版至少重复 3 轮。

指标至少看五组：质量（成功率/准确率/召回率）、合规（格式正确率/字段完整率/越界率）、稳定（三轮方差/失败波动率）、鲁棒（噪声输入和改写输入下的性能下降幅度）、效率（token 消耗/p50/p95 延迟/单位成功成本）。推荐门禁：质量提升至少 5%，合规和稳定不回退，成本增幅不超过预算阈值。

标准执行闭环：建立基线版本与指标快照 → 每次只改一个变量 → 先跑开发集淘汰劣化候选 → 再跑测试集验证泛化能力 → 满足门禁后再上线 → 上线后持续回收失败样本并更新评测集。

### 常见反模式与"魔法提示词"的工程解释

五类常见反模式：没定义目标就开始改；一次同时改很多变量；只看单次成功案例；把上下文问题误判成 Prompt 问题；迷信"魔法提示词"或"魔法 Token"。

先去神秘化：所谓"魔法提示词"不是咒语，而是某些写法恰好命中了模型训练分布里的高先验模式——任务框架被激活（如"以某角色视角分析"触发更稳定的任务模板）、结构先验被激活（如 JSON/XML/固定字段让模型进入更熟悉的结构化表达分布）、约束被显式化（字段、禁止项、验收项越清楚，输出熵越低）。但这些模式依赖模型版本、语言分布和任务类型，跨模型迁移时可能失效甚至反转，不能当长期硬编码规则。更稳的工程化用法：建候选模板库而非"神词崇拜库"，每个候选模板先做 A/B 和回归评测再决定是否入库，模板资产必须带元数据（适用任务、失效场景、模型版本、最近评测日期）。

本章最后落到一句话：稳定输出不是靠灵感，而是靠结构、评测和治理。只有当 Prompt 被放进模式、Schema、Eval 和模板资产这条链里，它才真正进入工程层。

## 六、从单次提问到可复用方法

### 为什么零散使用很难形成生产力

零散使用最大的损耗不是"问得慢"，而是每次都从零开始。没有沉淀，就没有规模化复用；没有复用，效率提升就只能停留在个人即时体验层面。

### 高价值任务的识别方法

优先沉淀四类任务：高频任务、高耗时任务、高错误成本任务、高协作摩擦任务。

**高价值任务识别矩阵**（横轴看频率，纵轴看错误成本）：
- 右上角（高频+高风险）：优先 SOP 化，最值得沉淀方法（例：发布前检查、线上故障复盘）
- 高频+低风险：优先模板化，适合先固化模板（例：需求拆解、周报生成）
- 低频+高风险：谨慎试用，先保守再自动化
- 低频+低风险：即时增效区，适合先试 Prompt

### 将高频任务抽象为标准输入与标准输出

这是从"经验"到"方法"的关键一步。以"接口变更评审"为例：标准输入（接口名称、改动目标、变更文件、已完成测试）；标准输出（变更摘要、风险清单、验证建议、是否可发布）。

### 将有效提问沉淀为模板

模板的价值不是复制 Prompt，而是把经过验证的写法固定成可复用入口：
```
任务类型：需求拆解
输入：需求描述 / 约束条件 / 非目标项
输出：1. 需求理解 2. 实现方案 3. 风险 4. 验收建议
```

### 将经验从个人直觉转为显式方法

当经验只存在于个人脑中，它就是不可复用资产。只有写成模板、规则、脚本、Skill，才会变成团队资产。

### 模板不是文案，而是带元数据的资产

真正可复用的模板应带有基本元数据：适用任务、输入字段、输出格式、失效场景、适配模型、最近一次评测时间。示例：
```yaml
name: feature_breakdown_v2
applicable_for: [需求拆解, 技术方案初稿]
input_fields: [requirement, constraints, current_system]
output_format: markdown_sections
failure_cases: [缺少边界条件, 推荐方案无回滚]
model: gpt-5
last_eval: 2026-03-10
```
真正关键的转变是把模板从"个人记忆"升级成"可治理对象"。

### 从模板到函数：参数化与版本治理

当模板高频使用，下一步是把它从模板资产推进到函数化单元：固定输入结构、固定输出结构、让不同调用场景只变参数不变骨架、对版本做显式管理。示例：
```json
{"template_name": "review_summary", "version": "v3", "params": {"module": "payment", "change_goal": "add retry queue", "changed_files": ["a.go", "b.go"], "test_results": ["unit passed", "integration pending"]}}
```

本章最重要的不是模板长什么样，而是是否开始识别哪些经验值得被固定下来。能不能把一次成功变成下一次还能复用的方法，这是从零散提问走向体系化协作的分水岭。

## 七、Skill：把 AI 能力固化为可调用单元

### 什么是 Skill

Skill 可以理解为"可触发、可复用、带边界的能力模块"，不是普通 Prompt 的简单别名，而是面向一类任务的稳定执行单元。适合做成 Skill 的研发任务：代码阅读、风险审查、文档整理、发布前检查、图资产门禁、特定系统查询与同步。

### 先判定类型：不要把所有 Skill 都写成长提示词

Skill 至少有三种主形态：
- **指令型**：核心是约束与流程，适合高不确定任务，例如澄清、阅读、审查框架。
- **脚本型**：核心是可执行能力，适合重复且可标准化任务，例如扫描、校验、批处理。
- **编排型**：核心是阶段门禁与责任边界，适合端到端交付链路。

创建前先判断四个维度：不确定性（需求是否频繁变化，是否需要大量判断）、重复性（动作是否高频重复，步骤是否稳定）、风险性（执行错误是否会引发较大代价）、外部依赖（是否强依赖外部系统实时状态）。判定结论：高不确定/低风险/低外部依赖 → 优先纯提示词或指令型 Skill；低不确定/高重复/中高风险 → 优先脚本化或脚本型 Skill；强外部依赖且需要真实最新数据 → 接入 MCP。同一个 Skill 可以是混合结构，但必须有主形态，否则职责会漂移。

### 合格 Skill 的标准结构

一个合格的 Skill 不是只有 SKILL.md，更稳的结构是：
```
skill-name/
├── SKILL.md
├── scripts/
├── references/
├── assets/
└── agents/openai.yaml
```
分工：SKILL.md 写清 name、description、阶段流程、门禁、失败策略；scripts/ 承接高频、重复、可确定的动作，要求参数清晰、幂等优先、退出码明确；references/ 放"只在需要时加载"的长文档，避免主说明膨胀；assets/ 放模板和产出素材，保证结果格式一致。判断准则：会被读但不一定每次都读，放 references/；会被用但不应塞进上下文，放 assets/。

### 什么时候在 Skill 中引入 MCP

需要真实源而不是口述上下文时引入（如设计实现依赖 Figma 节点与变量，任务推进依赖 Notion/Jira/GitHub 最新状态，或审查依赖远端系统事实数据）。不建议引入 MCP 的场景：仅本地文件就能完成、任务收益不足以覆盖接入与维护成本、团队尚未定义权限与审计边界。引入顺序：先做本地流程稳定 → 再接入单一高收益 MCP → 最后再扩展多系统联动。

### Skill 上线前验收清单

触发语句是否精确，能否和相近 Skill 区分；是否有明确边界，哪些事做哪些事不做；关键路径是否有可执行门禁；输出是否可复核，有记录、有证据、有失败说明；是否具备迭代入口，失败样本如何回流。

### Skill 与普通 Prompt 的差异

| 对比项 | 普通 Prompt | Skill |
|---|---|---|
| 目标 | 解决一次问题 | 解决一类问题 |
| 约束 | 临时写在对话里 | 固化在资产里 |
| 流程 | 人自己记 | 文档与脚本共同约束 |
| 复用 | 弱 | 强 |
| 协同 | 通常单轮 | 可参与多阶段链路 |

Skill 不是模板而是可组合能力单元：单个 Skill 解决单点问题，多个 Skill 可以在一次任务里被自动协调，Agent 可以按语义选择 Skill 而不是只能人工逐步指定。模板给的是固定路径，Skill 给的是能力单元。

**Skill 设计画布**（要能长期复用，必须把场景、输入、约束、流程和回退写清楚）：场景（这类任务什么时候触发，典型使用者是谁）、输入（要提供哪些事实/参数/上下文，哪些输入缺失会阻断执行）、约束（哪些动作绝对不能做，哪些门禁必须通过）、执行流程（1. 识别任务 2. 读取上下文 3. 执行动作 4. 自检门禁 5. 输出结果 6. 必要时回退）、输出与回退（最终产物是什么，如何判断成功，失败时如何安全退出）。

### AGENTS.md、CLAUDE.md 与 Skills 的配合

AGENTS.md 与 CLAUDE.md 本质上是同一类资产，都是项目级执行约束文件，负责补充当前项目的必备信息并指定项目级 SOP；Skills 负责把其中的具体步骤变成可执行手册。简化成一句话：AGENTS.md/CLAUDE.md 是项目级 SOP，回答"在这个项目里应该怎么做"；Skills 是操作级 SOP，回答"这个步骤具体怎么做"。配合顺序：先由项目级文件给出全局边界和门禁 → 再由 Skills 提供可执行任务手册和脚本能力 → 最后通过审查和提交类能力完成收口。反馈回规则与 Skill，形成"声明式约束 + 可执行约束"的完整闭环。

### 人机协作四个维度

看 AI 协作可以只看一个二维问题：人是否知道、AI 是否知道。四象限：人知道 AI 也知道（通用知识，补充优先级最低）；人不知道 AI 知道（可用于启发思路，但须经过项目约束和事实校验）；人不知道 AI 也不知道（未知区，需要补证据、查数据、做实验，不应直接决策）；**人知道 AI 不知道**（最关键区，应优先补给 AI——项目真实边界、团队约定与隐性规则、历史技术债与高风险点、当前迭代目标与非目标）。

### 本地 Skills 的入口、发现与管理

两类常见目录：Claude Code（项目级 `.claude/skills/<skill-name>/SKILL.md`，个人级 `~/.claude/skills/<skill-name>/SKILL.md`）；Codex（项目级 `.agents/skills/<skill-name>/SKILL.md`，个人级 `~/.agents/skills/<skill-name>/SKILL.md`）。目录选择原则：项目级解决团队一致性，个人级解决个人效率，先项目后个人、先稳定流程再扩展技能。本地库管理目标：技能可发现、来源可追溯、更新可回滚。

**发现渠道：先本地，再外部**。本地已有技能能覆盖的场景，不新增外部技能。外部发现分三层：第一层 CLI 与官方仓库检索（精确检索确认是否已有成熟能力）；第二层技能目录站点（按场景/热度/领域横向筛选）；第三层 GitHub 与社区清单（回到源码、Issue、更新时间和维护状态做最后审查）。推荐渠道：vercel-labs/skills（开放生态官方 CLI 仓库，`npx skills find`/`add`/`update`）、Skills.sh（开放技能目录）、SkillsMP（大规模 SKILL.md 聚合检索）、Claude Skills Directory（适合找与 MCP 连接器协同的技能入口）、awesome-claude-code（社区仓库级发现）。

**新 Skill 引入门禁**：来源可信（优先官方仓库、活跃社区仓库、可审计代码仓库）、结构合格（SKILL.md 描述清晰，必要时有 scripts/ 和测试样例）、维护可用（近期有更新，安装说明完整，有基本维护响应）。更稳的做法：先在单一场景灰度使用，连续通过门禁后再升级为团队默认可选技能，连续低命中/质量波动/维护停滞的技能进入归档候选。

### 生命周期与治理

Skill 更适合用生命周期去管理，而不是"装上就不管"：新建或引入（明确用途、边界、来源和结构是否合格）→ 小流量验证（观察命中率、稳定性和副作用）→ 评估与升级（收益稳定、门禁通过后升级为默认链路）→ 迭代或归档（持续有效的做更新，低命中/失效/维护停滞的做降级、归档或移除）。治理方式建议触发式而非固定周期：当返工轮次上升、交付链路频繁卡点、质量波动变大、某类任务开始高频重复时，触发一次盘点，决定是补规则、改 Skill、拆分 Skill，还是直接归档。

### 为什么 Skill 会越用越值钱

Skill 真正稀缺的地方不是写出了一份说明，而是它会随着使用不断积累经验。相同模型底座之上，真正拉开差距的往往不是模型本身，而是有没有持续沉淀上下文资产库。Skill 体系在使用中不断进化：负反馈闭环（执行偏差 → 人类纠正 → 更新规则或 Skill → 后续任务继承修正）；正反馈闭环（多次成功 → 抽象经验 → 形成新 Skill → 扩大能力池）。这就是"复利系统"的真实含义：不是模型自动变聪明，而是高质量经验被持续转成长期资产。

本章最终落到一个务实判断：Skill 的价值不在于写出来，而在于它能不能被稳定触发、被验证、被复用、被治理。

## 八、MCP、工具调用与外部能力接入

### 什么是 MCP

MCP 可以简单理解为"把模型连接到真实系统的标准方式"。它让 AI 不再只依赖人工转述，而是能接到代码库、文档系统、数据库和工具链。MCP 在研发工作中的意义：降低上下文失真、提升工具调用稳定性、为执行闭环提供真实输入。一旦 AI 能稳定连接这些系统，工作方式开始变化：从"问完再手动查"变成"边查边做"；从"拿建议"变成"拿结果"；从"单轮聊天"变成"执行工作流"。

### 工具调用、上下文补全与执行闭环

没有上下文补全，工具调用只能做零散操作；没有执行闭环，工具调用就无法形成生产力。真正成熟的方式，是把读取、判断、执行、验证、记录放在同一条链路里。

### 为什么结构化输出是 Agent/MCP 的前提

如果输出不能被稳定解析，后续工具就无法可靠消费。结构化输出是 Agent 和 MCP 从"能玩"到"能落地"的硬条件之一。

**案例：Anthropic MCP**——官方定位是开放标准而非单一插件："连接数据源、业务工具和开发环境，让 AI 在真实系统里工作"，用于解释"聊天式 AI"和"工具化 AI"之间的分水岭。

### Skills、MCP、Agents 是并列三支柱

不要把三者混成一个概念，它们解决三个不同问题：Skills 负责"怎么做得稳"；MCP 负责"能不能接到真实系统"；Agents 负责"多步骤、多角色任务怎么被编排完成"。如果这三者不分，团队就很容易停留在"工具很多，但闭环很弱"的状态。

### 何时用提示词、何时用脚本、何时接 MCP

用提示词：任务高不确定、需要分析和澄清、失败成本可控。用脚本：任务高频重复、可标准化、要求确定性和可审计。用 MCP：必须读取真实状态，人工转述会造成失真或时效问题。顺序：先用提示词确定边界 → 再用脚本固化重复动作 → 最后在关键真源链路接入 MCP。三个例子快速判断："帮我梳理这个新需求的风险和方案"→ 先用提示词；"每次发布前都要检查 12 个固定项"→ 优先脚本；"我要确认 Jira 状态、GitHub 变更和设计稿最新版本"→ 需要 MCP。

本章最后要带走的不是"多接几个工具"，而是三件事：接入真实上下文、形成执行闭环、把权限和边界设计清楚。

## 九、构建个人 AI-SOP

### 先立项目级"操作系统"

AI-SOP 的起点不是多加几个工具，而是先有一份项目级执行约束（AGENTS.md 或 CLAUDE.md），定义范围、目录、阶段门禁、命令规范、架构约束、输出标准与协作约定。必须先回答四个问题：In Scope 是什么、Out of Scope 是什么、关键门禁是什么、交付结果如何记录和回流。至少统一四类信息：范围（当前阶段做什么不做什么）、结构（目录/分层/模块职责和改动边界）、执行（允许使用哪些命令/脚本/工具和验证方式）、收口（审查/提交/归档和回流规则）。如果团队同时使用多种 AI 编码环境，关键不是维护两套不同方法，而是维护同一套规则在不同入口下的一致表达。

### 用 8 段结构定义主流程

需求澄清 → 结构拆解 → 模块设计 → 风险识别 → 生成 → 自检 → 约束回归 → 文档化。这套骨架的价值在于先定义流程，再决定每一步沉淀成什么（哪些只需要 Prompt 和 Pattern、哪些要固定 Schema、哪些要进入 Eval、哪些要沉淀成模板资产、哪些要升级成 Skill 或脚本、哪些需要接入 MCP）。

可映射到不同技术栈，例如后端场景：需求澄清先锁接口目标、数据边界和非目标能力；结构拆解按接口层、服务层、数据层和模型层划分职责；风险识别优先鉴权、一致性、幂等、回滚和兼容风险；约束回归对不符合分层边界和契约的改动一律回收。前端场景类似地按页面职责、状态边界、交互目标展开。

以"新增一个导出报表功能"为例走一遍 8 段流程：1. 需求澄清（导出哪些字段，给谁用，数据范围）2. 结构拆解（页面入口、导出接口、任务执行方式、文件存储方式）3. 模块设计（控制器/服务/导出模板/权限校验分别放哪层）4. 风险识别（大文件超时、权限越界、导出字段缺失）5. 生成（先产出接口和导出逻辑初稿）6. 自检（校验导出结果、性能和权限边界）7. 约束回归（确认没有破坏原有分层和接口契约）8. 文档化（把使用方式、限制条件和回滚方式写清楚）。

### 把流程映射成可执行闭环

闭环核心不是"有五个 Skill"，而是这五个 Skill 的执行顺序、目录结构和门禁能否复用。一条稳定的执行链会收敛成五段能力顺序：1. 澄清类 Skill（Question）2. 阅读类 Skill（Read）3. 实现类 Skill（Work）4. 审查类 Skill（Review）5. 收口类 Skill（Submit）。把这五段顺序固定下来的意义是把一次任务的风险控制切成三段：前置控制（用澄清和阅读避免从错误前提起步）、过程控制（用实现环节的范围锁定、门禁和验证避免错误落地）、末端控制（用审查和收口避免问题带着证据不足直接发布）。

**AI-SOP 标准执行链**：需求澄清 → 上下文读取 → 实现执行 → 审查 → 提交收口 → 记录与反馈进入下一轮闭环（审查未通过则回流修正）。关键点：前置控制、过程控制、末端控制必须同时存在，AI-SOP 才能从"能用"变成"能稳定运行"。

五个 Skill 的实现顺序与目录结构拆解：

1. **澄清类 Skill**：接收模糊需求 → 拆成目标、边界、约束和待确认项 → 输出结构化结论，确认后再进入实现链路。目录：`question-skill/SKILL.md`。设计重点：纯指令型，不急着接脚本，先把不确定性压缩。

2. **阅读类 Skill**：识别语言与技术栈 → 识别入口、架构、模块和关键数据流 → 留下阅读摘要，让上下文不只停留在会话里。目录：`read-skill/SKILL.md`。设计重点：以指令型为主，关键不是"看过"而是"读完留痕"。

3. **实现类 Skill**：明确 In Scope/Out of Scope → 锁定允许改动的模块和文件 → 记录变更前状态，隔离无关脏改动 → 确认模块职责和改动落点 → 先给最小可行方案再进入实现 → 在职责范围内完成改动 → 运行注释、规范、测试、构建等门禁 → 记录验证结论与阻断原因 → 回看复杂度和结构是否失衡 → 生成本地执行记录。目录：`work-skill/{SKILL.md, scripts/gate-checks.*, references/backend|frontend/style-guide.md...}`。设计重点：这是主产出层，风险最高，规则层、执行层、流程层要同时到位。

4. **审查类 Skill**：读取变更事实与任务目标 → 确定审查模式和深度 → 输出分级问题、风险判断和证据说明。目录：`review-skill/{SKILL.md, scripts/review-gates.*, references/security.md, complexity.md}`。设计重点：重点不是"挑问题"，而是给出可复核证据。

5. **收口类 Skill**：读取审查结论 → 判断是否满足提交条件 → 生成提交、记录和归档信息。目录：`submit-skill/{SKILL.md, assets/summary-template.md}`。设计重点：负责把任务从"完成"变成"可追溯结束"。

这五类 Skill 的实现顺序也应固定：先做澄清和阅读 → 再做实现层 → 实现层稳定后再加审查 → 最后补齐收口层。本质是先把主流程跑通，再逐步把风险控制前置和收口闭环补齐。

### 五类能力层如何接入闭环

对齐层（接收模糊需求，拆成目标/边界/约束/待确认项）、认知层（识别语言/入口/架构/模块和关键数据流）、实现层（锁定范围，确认结构，先出最小可行方案再实现并完成验证门禁）、审查层（读取事实，做风险分级，补充复杂度/注释/安全证据）、收口层（先审查再决定提交编排，最后生成记录并完成归档）。关键点不是名字，而是每一层都要有清晰输入、动作、门禁和输出。

### 何时用纯提示词、何时上脚本、何时接 MCP

**纯提示词优先**：任务是澄清、分析、结构化表达；失败成本低，且可人工快速复核；不依赖外部实时状态。**脚本优先**：任务高频重复、可标准化；需要确定性、可复跑、可审计；人工执行容易漏项。**MCP 优先**：必须读取外部真实上下文；手工转述会导致信息损耗或时效失真；需要跨系统可追溯调用。组合策略：先用提示词确定边界 → 再用脚本固化重复动作 → 最后在关键真源链路接入 MCP。典型组合示例：先让 AI 列出"本次发布检查需要什么" → 把其中固定可执行的 8 项写成脚本 → 如果其中 3 项依赖真实工单状态、远端仓库和监控平台，再用 MCP 连接真源。

### 先跑最小可运行 SOP，再扩展到全链路

不要一开始就做全场景 SOP，先选一个高频场景跑通。最小可运行 SOP 至少要有：输入（目标清单、边界清单、非目标清单）、输出（实现结果、验证结果、风险台账、变更记录）、门禁（链路闭环通过、契约稳定、审查无阻断项）。它的意义不是"先做小一点"，而是先验证这条链是不是真的能跑。

示例场景"每次新增接口时的标准处理链"：输入（需求说明、接口契约、约束项）；流程（需求澄清 → 方案拆解 → 实现 → 测试 → 审查 → 文档）；输出（改动代码、测试结果、审查结论、接口说明）；门禁（接口契约无漂移、关键测试通过、无阻断风险）。

### 增强层按阶段接入，不做"工具堆叠"

按阶段接入：阶段 1 先找已有能力（发现本地是否已有 Skill，不重复造轮子）；阶段 2 把高频任务稳定下来（让提示词模板化，降低输出波动）；阶段 3 再扩并行和编排（多 Agent 任务才引入协同，不把复杂度前置）；阶段 4 把有效流程沉淀成 Skill（进入可复用资产库）；阶段 5 按需接外部系统（先验证收益，再固化为默认链路）。原则只有一句：**先跑通，再堆叠**。

### 让 SOP 可进化：持续迭代

不应按固定周期机械重做，更合理的是触发式迭代：当出现连续返工、质量波动或交付卡点时，做一次技能梳理决定增/并/废；当某条流程在多次任务中表现稳定时，升级为默认链路；当某条流程持续失效时，降级或归档。失败沉淀模板也很重要：问题现象、触发条件、影响范围、新约束、对应 Skill 或规则更新。

### AI-SOP 的资产装配图

AI-SOP 不是替代前面那些层，而是把前面那些层装配成系统：规则资产、模式资产、结构资产、评测资产、模板资产、Skill 资产、工具接入、复盘资产。

**个人 AI-SOP 的五层结构**（规则、模板、Skill、工具和反馈指标，共同组成可持续迭代的个人工作体系）：
1. **规则**：边界、风格、权限、质量门禁
2. **模板**：Prompt 模板、结构化输出模板、检查模板
3. **Skill**：场景化能力模块、脚本、参考资料
4. **MCP/工具**：连接仓库、文档、数据库、设计工具
5. **反馈指标**：通过率、返工轮次、交付时长、复用率

**AI-SOP 组件协同关系**（从任务输入到证据回流）：项目任务 → Agent 编排（任务拆解、步骤调度、上下文组织）→ API 推理能力 / MCP 工具接入 / Skill 方法模块 → AI CLI/终端执行（仓库操作、脚本执行、结果落盘）→ 证据与反馈回流（审查记录、规则更新、Skill 迭代）。关系说明：API 提供基础推理能力，是整个协作链路的能力底座；MCP 负责连接真实系统，Skill 负责固化稳定方法，二者共同支撑执行；没有证据与反馈回流，协作只会停留在一次性完成，无法沉淀为 AI-SOP。

### 用指标持续迭代 AI-SOP

基本观测指标：一次通过率、返工轮次、平均交付时长、Skill 复用率。若想和提示词评测体系打通，还可加：模板资产命中率、结构解析成功率、评测门禁通过率、回流样本增长率。没有这些指标，团队就只能感受到"好像快了一点"，却很难真正判断体系有没有变稳。

本章真正的结论：AI-SOP 不是一张流程图，也不是几条规则，而是一套能持续运行、持续观察、持续升级的工作系统。只要能跑通最小闭环，并且让经验不断回流，这套系统就会开始产生复利。

## 十、最终目标——设计自己的 AI 工作方式

真正的分水岭，不是谁会用更多工具，而是谁能把前面几章的方法链设计成自己的协作模式。如果今天学到的只是几个 Prompt 技巧，价值会很快衰减；如果学到的是一条从单次任务走向资产化、函数化和系统化的路径，它会变成能力复利。

### 建立个人 AI 能力成长路径

六步路径：先学会正确提问 → 再学会抽象可复用模式 → 再学会固定结构与评测 → 再学会沉淀模板资产与函数化单元 → 再学会沉淀 Skill → 最后设计自己的 AI-SOP。

30 天落地节奏：第 1 周固定 2 个高频 Prompt 模板；第 2 周给其中 1 个模板补 Schema 和验收标准；第 3 周把 1 个重复动作沉淀成 Skill 或脚本；第 4 周把这几个能力串成第一条个人 AI-SOP。

**个人 AI 能力成长路径**（从会问、会控，到会沉淀、会复用，最终走向"会设计自己的协作系统"）：L0 会问（知道怎么给背景、目标、约束）→ L1 会控（会要结构化输出和验收标准）→ L2 会沉淀（会做模板、Pattern、复盘）→ L3 会复用（会设计 Skill 和协作链路）→ L4 会设计（会构建自己的 AI-SOP 与工作方式）。课程结论：真正的目标不是"更会用 AI"，而是"更会设计自己的 AI 协作系统"。

### 下一阶段：Agent、自动化与更强的个人工作流

当完成前面的积累后，Agent、自动化、MCP 才真正有意义，否则它们只是复杂度升级而非能力升级。前面几层负责把单次成功变成稳定资产；Agent 和自动化负责把这些资产调度起来，在更长链路里运行。

### 长期护城河不是模型，而是上下文资产库

护城河不在基座模型本身，而在长期沉淀后的上下文资产库——具体体现在规则、Pattern、Schema、Eval 结果、模板资产、函数化单元、Skills、失败样本和任务编排结构里。模型会持续升级，但这些资产不会因为模型升级而失效，反而会在更强模型上得到更高兑现率。

### 从个人系统走向团队落地

演进路径：工具普及 → Prompt 与 Pattern 方法 → Schema 与 Eval 稳定化 → 模板资产与函数化沉淀 → Skill 沉淀 → AI-SOP 体系 → Agent 自动化。这也是为什么这门课不能只停留在"提问技巧"，而必须把工作流、门禁和资产沉淀一起讲清楚。

### 结课判断标准

四个变化：不再只追求"问出一个答案"，而会先设计任务结构；不再只收集 Prompt，而会沉淀模式、结构、模板资产、Skill 和评测方法；不再只关注单点效率，而会关注整条执行闭环；不再只把 AI 当工具，而会开始设计自己的协作系统。

## 十一、三类关键方法如何组成一套体系

- **提示词负责控制输入与输出**（对应第二、五章）：管任务定义与输出控制，重点是把单次 Prompt 提升到 Pattern、Schema 和 Eval。
- **Skill 负责固化高频动作**（对应第六、七章）：管高频动作固化，重点是把模板资产继续推进成函数化单元和可调用能力。
- **AI-SOP 负责串成完整闭环**（对应第八、九章）：管系统装配，重点是把规则、Skill、工具接入、评测和复盘真正串成闭环。

## 十二、附录：可直接复用的模板

**通用研发 Prompt 模板**：
```
背景：当前项目/模块/已知事实
目标：这轮希望完成的具体结果
约束：不能改什么，必须满足什么
输出：请按 Markdown/JSON/表格输出
验收：覆盖项、验证项、失败定义
```
示例：
```
背景：当前在支付模块排查退款重试失败问题，已知日志里出现 duplicate key 和 timeout。
目标：请判断最可能的原因，并给出修复方案。
约束：不要修改数据库 schema；不要假设外部支付渠道提供新能力。
输出：请按"判断/假设/修复/验证/回滚"输出。
验收：必须覆盖幂等、重试、回滚三项；不确定的地方要单独列出。
```

**Skill 设计模板**：
```
Skill 名称：
适用场景：
输入：
约束：
执行流程：
输出：
失败回退：
需要引用的脚本/文档：
```
示例：
```
Skill 名称：release-check
适用场景：发布前标准检查
输入：发布分支、变更摘要、测试结果
约束：不允许跳过关键检查；没有证据不能判定通过
执行流程：读取变更 -> 执行检查 -> 汇总风险 -> 输出结论
输出：检查结果、阻断项、建议动作
失败回退：发现阻断项时停止发布并返回修复清单
需要引用的脚本/文档：scripts/release_check.sh、references/release-gate.md
```

**个人 AI-SOP 画布**：我的高频任务是什么；哪些任务已经可以模板化；哪些任务应该升级为 Skill；哪些系统需要通过 MCP 连接；哪些指标用来判断我的协作方式是否在变好。建议补三组追问：哪些任务一旦出错返工成本最高；哪些步骤最适合交给脚本而不是继续靠人工记忆；哪些结果必须留下证据，否则下次无法复用。

**课后行动建议**：1. 先选一个高频任务，把 Prompt 写成固定模板 2. 再把这类任务抽象成 Pattern，并固定 Schema 3. 再选一个高错误成本任务，补上 Eval 和验收标准 4. 再把一个重复任务升级成模板资产、函数化单元或 Skill 5. 最后把这些能力串成自己的第一条 AI-SOP。如果要更容易执行，可以直接从三个任务里选一个开始：代码阅读摘要、需求拆解、发布前检查。

## 十三、结语

这门课真正想交付的，不是"怎么把 AI 用得更顺手"，而是"怎么把 AI 变成长期能力资产"。Prompt 可以带来一次成功，Pattern 可以带来一类成功，Schema 和 Eval 可以带来稳定控制，模板资产和函数化单元可以带来持续复用，Skill 可以带来稳定执行，AI-SOP 才能把这些层真正装配成持续复利的个人工作方式。
