## 一、你发现了什么问题或机会？

### 当前现状

敞亮科技在飞书上沉淀了大量组织知识——**8 个 Wiki 空间**、数十份文档、多个群聊——但新人入职时面临严重的信息断层：

| 痛点 | 具体表现 |
|------|----------|
| 📂 **文档分散** | 8 个 Wiki 空间各自独立，跨空间搜索能力弱，新人不知道从哪里开始 |
| 🔗 **培训与资料分离** | 培训计划表只有任务名称，没有对应的文档链接 |
| 💬 **IM 知识丢失** | 群聊有大量决策上下文，但新人加群晚，看不到历史讨论 |
| 👨‍🏫 **导师依赖严重** | 非显性知识全凭口述——"为什么选 OpenIM""产品是怎么演进到今天的" |
| 🔐 **权限信息缺失** | 新人不知道要申请哪些权限，每次都是遇到报错才临时补 |

### 真实案例

fly（一笔），前端开发，2026 年 6 月 4 日入职。为理解智能客服产品全貌，他需要：

- 翻遍 8 个 Wiki 空间
- 找到并阅读 10+ 篇相关文档
- 向 5+ 位同事反复请教
- **花 3 天时间才拼凑出完整业务全景**

### 更深层的机会：AI 能力基线参差不齐

公司明确鼓励 AI 应用（AI LAB 有专项指引），但员工 AI 使用水平依赖个人探索，差距在入职第一周就已拉开：

| 层级 | 工具 | 特征 | 公司内占比（估） |
|------|------|------|:--:|
| **L1 聊天级** | 豆包、ChatGPT 网页版 | 单次问答，无工程上下文 | ~60% |
| **L2 项目级** | Claude Code、Cursor | 代码生成、有工程上下文 | ~35% |
| **L3 自主规划级** | lark-cli + Agent | AI 主动理解上下文、规划工作 | <5% |

**公司缺少让新人从第一天就建立 L3 级 AI 协作心智的机制。**

## 二、你的解决方案是什么？

**"敞亮伴侣"** —— 基于 lark-cli 的桌面端 AI 入职导航 Agent。

### 产品形态

- **桌面应用**（Windows/macOS），对话式交互界面
- 预集成 lark-cli 全权限（文档、IM、Wiki、Sheet、搜索），开箱即用
- 无需服务端——知识索引和检索全部本地运行（LanceDB），数据不出本机

### 两大知识域，分层组织

| 知识域 | 内容 | 数据来源 |
|--------|------|----------|
| 🏢 **公司通用** | 企业文化、规章制度、薪酬福利、HR/行政/财务流程、新人培训计划 | 公司规章制度 Wiki、敞亮文化 Wiki、飞书表格 |
| 💼 **部门业务** | 产品背景与演进、技术架构、当前版本需求、团队分工、技术选型 | 部门 Wiki、IM 群聊、技术文档、需求文档 |

### 两种交互模式：Pull + Push

| 模式 | 行为 | 示例 |
|------|------|------|
| **Pull（主动查询）** | 用户向 Agent 提问 → Agent 判断知识域 → 搜索相关文档 → 生成回答并附引用来源 | "我们部门用的什么技术栈？""公积金比例是多少？" |
| **Push（主动推送）** | 每日定时：读取培训计划 → 对比进度 → 关联当前需求文档 → 检索群聊讨论 → 生成今日建议 → 系统通知推送 | 📬 "本周你的任务涉及座席端融合方案，建议先看这 3 篇文档，产品群昨天刚讨论了相关技术选型" |

### 技术本质

一个预配置好飞书全上下文的桌面端 Agent 壳。底层通过 lark-cli 读写飞书全数据源，上层是对话式 UI + 系统通知推送。

```
桌面应用 (Electron + Vue 3)
├── 对话 UI（Pull：用户主动提问）
├── Push 通知（系统通知栏）
├── Agent 核心
│   ├── 对话管理 + 上下文调度
│   └── RAG 知识检索（LanceDB 本地向量库）
└── lark-cli 数据接入层
    └── Doc | Wiki | IM | Sheet | Search
```

## 五、为什么你的方案与众不同？（核心创新点）

### 1. 不只是 RAG 问答机器人——是全飞书数据源的 Agent

市面上的 AI 问答助手大多只吃几篇上传的 PDF。敞亮伴侣基于 lark-cli，能实时读取飞书 **IM 消息、Wiki 空间、在线文档、电子表格、搜索结果**五大数据源——这是飞书生态内 AI 应用的能力天花板。

### 2. Push 主动推送——从"等人来问"到"主动告知"

目前所有 AI 助手都是被动问答模式。敞亮伴侣通过读取新人培训计划、群聊讨论动态、当前版本需求，**主动判断**新人该关注什么，在系统通知栏推送——把 AI 从工具变成搭档。

### 3. AI 能力基线植入——产品即教练

敞亮伴侣本身就是一个 **L3 级 AI Agent**。新人在使用它的过程中，自然理解三件事：
- AI 可以**主动获取上下文**（非被动问答）
- AI 可以**跨数据源整合信息**（飞书全数据）
- AI 可以做**工作规划和经验沉淀**

不是在培训里讲 AI 怎么用，而是让新人第一天就和 AI 搭档工作。

### 4. 双重上下文 + 零服务端

公司通用制度 + 部门业务知识分层组织，新人一键切换。全部索引和检索在本地运行（LanceDB），无需服务端，保障数据隐私，零运维成本。

## 六、如何验证它是否可行？

### 试点方案

- **范围**：智能客服产品开发组，伴随下一位新人入职
- **方式**：A/B 对比（使用敞亮伴侣的新人 vs 传统入职流程的新人）
- **周期**：新人入职第一周

### 验证指标

| 指标 | 测量方式 | 目标 |
|------|----------|:--:|
| 自主信息获取率 | 新人入职第一天可回答的常见问题覆盖率 | **≥ 80%** |
| 导师问答频率 | 基础性问题咨询次数对比 | 下降 **≥ 50%** |
| 业务理解深度 | 第一周结束时能否独立画出产品架构图 | **能** |
| 新人满意度 | 入职体验评分 | **≥ 4.5/5** |

### 落地路径

```
试点达标 → 推广至 LP2-HG 事业部所有新人
       → 二期：知识域模块化，各部门自主配置
       → 三期：与培训计划表双向联动（Agent 自动更新完成状态）
```
